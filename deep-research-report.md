Your recent push already implements an edge function to generate and upload batch files to an SFTP endpoint, which is a good foundation for “direct‑to‑state” billing. To complete full automation, however, the system must (1) securely handle provider credentials, (2) transmit files via SFTP to MN‑ITS, (3) ingest 999 and 835 responses, and (4) update claim statuses for each tenant. Below is a technical and operational plan aligned with those findings.

**Technical recommendations**

- **Secure credential storage** – Each provider will need MN‑ITS SFTP credentials (user ID and password/certificate) to submit claims. These must not be stored in plain text. Use a secrets store such as Supabase Vault, AWS Secrets Manager or environment variables with envelope encryption. Generate a unique encryption key per `org_id` (multi‑tenant key management) and use that to encrypt the credentials at rest in your database. Your SaaS should expose a UI/API for providers to enter and rotate these credentials; store only the ciphertext. When the edge function runs, it retrieves and decrypts the credentials in memory, connects to MN‑ITS and uploads the batch; then it discards the plaintext credentials immediately afterwards.

- **SFTP worker** – Build a background function (for example, a Supabase Edge Function or a serverless cron job) that reads pending claims grouped by organization, assembles them into a HIPAA‑compliant 837P file, and transmits it via SFTP to the MN‑ITS server using the provider’s credentials. The function should run on a schedule (e.g., nightly) or be triggered manually. Use unique file names incorporating the provider’s Medicaid ID and date. Log each transmission in a `claim_batches` table with the `org_id`, batch ID, timestamp, and status.

- **Response parser** – After submitting claims, the system must poll or receive 999 acknowledgement and 835 remittance files from MN‑ITS. Implement a parser that reads these files, associates them with the correct batch and provider, and updates each claim status to `accepted`, `paid` or `denied`. Store parsed fields (e.g., payment amount, denial reason) in a `claim_responses` table keyed by `claim_id` and `org_id`. Trigger notifications or dashboards so providers can see payment status.

- **Tenant‑scoped storage and encryption** – All generated files and response files should be stored in tenant‑scoped buckets (e.g., `/billing-files/{org_id}/`), with server‑side encryption. Row‑level security policies should ensure that each organization can access only its own claims, batches and stored credentials. Use per‑tenant encryption keys to protect file contents; this adds defense in depth.

- **Sandbox and testing** – Before going live, obtain MN‑ITS sandbox credentials for each provider and run end‑to‑end tests: generate 837P files with realistic waiver claims, transmit via SFTP, ensure that 999 and 835 files are parsed correctly and statuses update automatically. Validate that multi‑tenant isolation holds and that a misconfigured credential cannot access another provider’s data.

Sign BAAs with each provider, clarifying responsibilities for security, breach reporting and data handling. Also sign a BAA with Supabase (or your infrastructure provider) if it stores PHI. Ensure SOC 2‑type controls (audit logs, access reviews, incident response) are in place.
